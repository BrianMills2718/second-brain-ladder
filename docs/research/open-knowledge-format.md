# Open Knowledge Format (OKF) — Google's markdown-files-with-YAML convention for feeding curated context to AI agents

- **URL / source / date / who's behind it:**
  - Marketing post: [How the Open Knowledge Format can improve data sharing](https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing), Google Cloud Blog, 12 June 2026, by Sam McVeety & Amir Hormati (Google Cloud Data Cloud tech leads).
  - Spec + reference code: [GoogleCloudPlatform/knowledge-catalog/okf](https://github.com/GoogleCloudPlatform/knowledge-catalog/tree/main/okf) (`SPEC.md`, Apache 2.0, version 0.1).
  - **Behind it: Google Cloud alone.** No consortium, no W3C/standards body. It is a Google-published "open specification," not a chartered standard. Independent coverage: [MarkTechPost](https://www.marktechpost.com/2026/06/16/google-cloud-introduces-open-knowledge-format-okf-a-vendor-neutral-markdown-spec-for-giving-ai-agents-curated-context/), critical take by [Marc Bara (Medium)](https://medium.com/@marc.bara.iniesta/googles-new-format-for-agent-context-a-standard-or-just-a-folder-82fb21d92041).

- **What it is:**
  OKF v0.1 is a lightweight convention for representing organizational knowledge as **a directory of Markdown files with YAML frontmatter**. Each `.md` file is one "concept" (a table, dataset, metric, playbook, runbook, API, etc.); files cross-link with ordinary Markdown links, turning the folder into a loose graph. Its goal is to formalize the ad-hoc "LLM wiki" pattern into a portable bundle that any agent can consume without translation — solving the problem that context for AI today is scattered across catalogs, wikis, code comments, and people's heads. It is explicitly *not* a knowledge-representation formalism; it is a packaging/interchange convention for agent context.

- **Key design points:**
  - Reserved frontmatter fields are minimal: only `type` is required; `title`, `description`, `resource` (a URI), `tags`, `timestamp` are recommended. Producers may add custom keys; consumers must tolerate unknown keys and broken links.
  - Bundle conventions: optional `index.md` (directory listing / progressive disclosure) and `log.md` (change history); everything else is a concept doc.
  - **Graph model is deliberately weak:** links are *untyped directed edges* — "the kind of relationship is conveyed by the surrounding prose, not by the link itself." No typed predicates, no axioms, no reasoning.
  - **Relation to existing standards:** The spec does **not** build on or even mention RDF, JSON-LD, schema.org, property graphs, or GQL. It is positioned against RDF/OWL by contrast: the blog frames those early-2000s semantic-web standards as more expressive (typed relations, formal axioms, reasoning) but heavyweight (schema registries, specialized tooling, expertise), and pitches OKF as the pragmatic, LLM-era alternative.
  - **Novel vs repackaging:** Mostly repackaging. It is essentially the Obsidian / "docs-as-code" / metadata-as-code pattern (Markdown + frontmatter + wikilinks) given a name and a one-page spec. The novelty is purely social: a shared filename/field convention so different producers and consumers can interoperate, plus Google's reference agent + HTML visualizer + sample bundles (GA4, Stack Overflow, Bitcoin).

- **Relevance to a "second brain" curriculum:**
  - **Tangential as a foundational standard.** OKF is aimed at *enterprise* data-sharing and *agent* context (BigQuery tables, metrics catalogs, runbooks) — not at an individual building a personal knowledge base. A learner does not need to "know OKF" the way they should understand linking, atomic notes, or graph structure.
  - **But the underlying pattern is exactly the second-brain pattern** — Markdown + YAML frontmatter + untyped wikilinks forming an emergent graph is precisely how Obsidian/Logseq/Foam work. So OKF is best used in a curriculum as a *case study*, not a topic: it validates the "plain-files-as-graph" approach and illustrates the **portability/interchange lesson** worth teaching — keep knowledge in plain text with light, conventional metadata so it survives tool changes and can be consumed by agents and humans alike. That lesson (own your data, avoid proprietary lock-in, structure-by-convention) transfers directly; the OKF spec itself does not need to be on the syllabus.
  - Contrast point for teaching: OKF's *untyped* links vs. RDF/property-graph *typed* relations is a clean way to show the spectrum from "lightweight personal notes" to "formal knowledge graphs" — useful for explaining the tradeoff a second-brain builder faces.

- **Caveats:**
  - **Vendor post.** The blog's "improve data sharing / interoperability" framing oversells substance. Independent analysis (Bara) distinguishes **structural** interoperability (folder + Markdown + YAML — which OKF achieves) from **semantic** interoperability (shared vocabulary — which it punts on). Two bundles can be fully conformant yet share no common vocabulary: `type` values are unregistered (one writer says "BigQuery Table," another "table"), and links carry no semantics.
  - **Immature (v0.1) and internally inconsistent at launch:** Google's own reference parser expects ~4 fields while the spec mandates only `type` — the "required surface" isn't settled.
  - **"Open" is qualified.** The format and license (Apache 2.0) are genuinely open and vendor-neutral, but there is **no independent governance** — it lives in a GoogleCloudPlatform repo, and the demonstrated ecosystem is Google-shaped (Gemini produces it, BigQuery is the source, ingestion via Google's Knowledge Catalog). Adoption beyond Google is unproven as of mid-2026.

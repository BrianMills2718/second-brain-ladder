# Operating & maintaining a knowledge system — the day-2 "run-and-maintain" layer most resources skip

- **Scope:** keeping a knowledge/second-brain system *correct, current, and affordable in use over time* — the OPERATE layer of the learner-empowerment ladder (understand → decide → plan → implement → **operate**). Standing a system up once is the demo; running it for your own goal over months, as sources change and the store drifts, is the actual skill. This note is neutral across the three paradigms in the README (symbolic KG, GraphRAG, LLM-wiki/agentic notes): the *operations* problems are largely shared, and where a paradigm changes the cost, that is called out.

## Maintenance — the hard part

The recurring finding across every paradigm: **extraction/indexing is easy; keeping the store coherent over time is the 80%.** Karpathy's own framing is that "the tedious part of maintaining a knowledge base is not the reading or the thinking — it's the bookkeeping" (`karpathy-gist.md`), and the karpathywiki plugin spends 20+ modules on lint/dedup/contradiction machinery vs. a handful on extraction (`karpathywiki-obsidian.md`). The concrete sub-problems:

- **Incremental ingestion / re-indexing.** Two strategies: full re-index (simple, but compute-heavy and can cause downtime — fine for small corpora, schedule as an off-hours cron job) vs. incremental update (process only what changed: added/updated/deleted docs). Production systems prefer incremental, with an occasional full rebuild. Standard pattern: a **document registry** mapping doc-ID → chunk/vector-IDs with version numbers; on update, delete old chunks, re-chunk, re-embed only changed material ([Particula](https://particula.tech/blog/update-rag-knowledge-without-rebuilding), [apxml](https://apxml.com/courses/optimizing-rag-for-production/chapter-7-rag-scalability-reliability-maintainability/rag-knowledge-base-updates)).
- **Idempotent re-ingest.** The single most transferable pattern: **content-hashing**. Compute a hash per document (or per chunk); if it matches the registry, skip — most "updates" are metadata-only ([Arpit Bhayani, *What Matters in Production RAG*](https://arpitbhayani.me/blogs/rag-production/)). This is exactly karpathywiki's "Smart Batch Skip," which also gates a pre-ingest validation step so weak/local models don't hallucinate entities from empty files (`karpathywiki-obsidian.md`). Re-ingesting a source should *merge* into existing pages, not duplicate them.
- **Entity dedup & conflict resolution.** Re-ingestion constantly re-encounters the same entities under variant names. Graphiti's approach is a two-tier resolver: a deterministic **MinHash + LSH** fast path, with an **LLM fallback** for hard cases, scoped to the relevant entity pair to bound search cost; then edge dedup and contradiction resolution per episode ([Neo4j/Graphiti](https://neo4j.com/blog/developer/graphiti-knowledge-graph-memory/), [Zep paper](https://arxiv.org/html/2501.13956v1)). karpathywiki forces every page to carry ≥1 **alias** so a tiered semantic detector can catch cross-language dupes ("CoT" vs "思维链") and merge while preserving aliases (`karpathywiki-obsidian.md`).
- **Stale & contradictory facts.** New sources contradict old pages. The LLM-wiki answer is an explicit **lint pass** detecting contradictions, stale claims, orphans, and missing cross-refs, moving each contradiction through a state machine (`detected → review-passed → resolved/unresolved`). The graph-memory answer is **invalidation rather than deletion** — mark the old fact expired and keep it.
- **Temporal validity (bi-temporal).** Graphiti's distinctive feature: track **two timelines** — *valid time* (`valid_at`/`invalid_at`: when the fact was true in the world) vs. *transaction time* (`created_at`/`expired_at`: when the system learned it). This lets a fact recorded today retroactively describe a past event, and lets you answer "what did we believe on date X" vs. "what was actually true on date X" ([Graphiti](https://neo4j.com/blog/developer/graphiti-knowledge-graph-memory/), [Zep paper](https://arxiv.org/pdf/2501.13956)). It is the principled alternative to overwriting facts and losing history — directly relevant to a second brain that accumulates over years.
- **Dead links / orphans (the LLM-wiki problem).** A link graph rots: pages get renamed, references go stale, new pages aren't linked in. karpathywiki's "Smart Fix All" runs repairs in causal order — fill aliases → merge dupes → fix dead links → link orphans → expand empty pages (`karpathywiki-obsidian.md`). This class of problem is specific to the human-readable navigable-notes paradigm and is *the* maintenance burden Karpathy's `lint` operation exists to absorb.

**Why harder than the build:** the initial build is a one-shot batch over a clean snapshot; maintenance is a *continuous, online* process over a moving target where every update can corrupt invariants (duplication, contradiction, dangling links, embedding-model drift). GraphRAG makes this acute: adding documents to a community-summarized graph can require **rebuilding from scratch**, which is "impractical for dynamic environments" ([articsledge](https://www.articsledge.com/post/graphrag-retrieval-augmented-generation)) — a key reason incremental-friendly designs (LightRAG, Graphiti) and the append-merge LLM-wiki pattern exist.

## Evaluation in production

You cannot maintain what you don't measure; "it feels fine" is not an operating discipline.

- **Golden / regression sets.** Curate 30–50 real or realistic queries reflecting your main use cases; treat them as regression tests that catch quality drops *before* they ship ([Braintrust](https://www.braintrust.dev/articles/what-is-rag-evaluation)). Run them on every index update, embedding-model change, or corpus-wide change.
- **RAGAS & faithfulness checks.** RAGAS is the popular open-source RAG-eval framework; its key property is a **reference-free** design (no gold answers needed), which suits dynamic personal KBs. **Faithfulness** decomposes an answer into atomic claims and verifies each against retrieved context (1.0 = nothing fabricated) — the front-line hallucination check (`rag-fundamentals.md`, [articsledge RAGAS](https://www.articsledge.com/post/retrieval-augmented-generation-assessment-system-ragas)). A cheap production variant: LLM-as-judge over top-k chunks scoring faithfulness + relevance, sampling ~1% of traffic + 100% of user-flagged responses ([Arpit Bhayani](https://arpitbhayani.me/blogs/rag-production/)).
- **Drift detection.** Monitor faithfulness over time; a sudden drop usually means an index update introduced bad chunks or a model update changed generation. Evaluate continuously where telemetry exists, weekly for baselines, and immediately on big changes ([n-ix](https://www.n-ix.com/rag-evaluation/)). Common split: RAGAS/DeepEval for offline regression; Phoenix/LangSmith for production trace logging and drift.
- **Human-in-the-loop & gating updates on eval.** Tie this to the methodology's **gate** idea (CLAUDE.md: judge validated on a frozen `prompt_eval` set before it gates any achievement; targets false-pass ≤5%, false-fail ≤15%). The operating analogue: an ingestion/re-index run is *gated* on passing the regression set, and contradiction-resolution is escalated to human review (karpathywiki's `detected → review-passed`) rather than silently auto-merged. Build the eval harness **early**, not after problems appear.

## Cost, latency, observability

- **Indexing vs. query token costs.** These are separate budgets. GraphRAG indexing has historically run **$20–50 per million corpus tokens** for LLM entity/relation extraction; vector RAG is ~$0.001/document for one embedding pass ([casys.ai](https://casys.ai/blog/graphrag-vs-vectorrag), [TianPan](https://tianpan.co/blog/2026-04-17-graphrag-vs-vector-rag-knowledge-graphs)). Query cost also diverges: one study reports GraphRAG using ~610k tokens per retrieval vs. <100 for LightRAG (a ~6,000× spread). LLM-wiki ingestion is itself an LLM-heavy write-time cost — karpathywiki exposes granularity tiers (≈5→500 items/source) precisely to trade depth against token spend (`karpathywiki-obsidian.md`).
- **Caching & model tiering.** Cache embeddings (hash-skip unchanged content — "lazy re-embedding"), cache query results, and route by question type: simple lookups → cheap vector path, multi-hop/global → expensive GraphRAG path (hybrid routing). AI gateways (Helicone, Portkey) add caching/routing/cost-tracking with minimal code ([LangChain obs tools](https://www.langchain.com/resources/llm-observability-tools)).
- **Tracing / monitoring.** Tracing is the most important LLM-ops tool: capture per-request prompt, response, **token usage, latency, tool/retrieval steps**. Structure as nested spans (embed → vector search → rerank → prompt assembly → generation), tagging each retrieval span with the **index version** so quality drops correlate to specific deployments ([Arpit Bhayani](https://arpitbhayani.me/blogs/rag-production/)). **Langfuse** (open-source, MIT core, self-hostable) and **LangSmith** (managed, P50/P99 latency, cost breakdowns, error rates) are the reference tools; the 2026 trend is fusing observation with evaluation — scoring outputs and alerting on quality drift, not just logging ([Langfuse](https://langfuse.com/docs/observability/overview), [ZenML compare](https://www.zenml.io/blog/langfuse-vs-langsmith)).
- **When GraphRAG indexing cost is / isn't worth it.** Worth it for genuinely multi-hop, global "sense-making" questions over a corpus. Not worth it for simple lookups, for frequently-changing data (full-rebuild penalty), or at very large scale where gains plateau (~5–15M tokens). Mitigations exist: **LazyGraphRAG** claims GraphRAG-comparable accuracy at ~0.1% of extraction cost; LightRAG targets ~1/100th cost ([Microsoft via casys.ai], [ragdollai LightRAG](https://www.ragdollai.io/blog/lightrag-vector-rags-speed-meets-graph-reasoning-at-1-100th-the-cost)). The **embedding-model lock** is the sleeper cost: every stored vector is tied to its model, so switching models means re-embedding the whole index — store model name+version in chunk metadata, assert query-model == index-model, migrate via a shadow index ([Arpit Bhayani](https://arpitbhayani.me/blogs/rag-production/)).

## Update path & governance

- **Provenance.** Track where each fact came from and that it hasn't been tampered with: source-doc IDs, index timestamps, document versions, optionally cryptographic signatures; flag when source material changes ([nstarx](https://nstarxinc.com/blog/the-next-frontier-of-rag-how-enterprise-knowledge-systems-will-evolve-2026-2030/)). Graphiti's `EpisodicNode` retains the raw input each fact was extracted from — provenance by construction. In the LLM-wiki paradigm, the citation links *are* the provenance, and the append-only `log.md` is the audit trail.
- **Versioning the store.** Index aliases pointing to validated, overnight-built indexes; atomic alias swap for zero-downtime cutover; retain old indexes for rollback ([Arpit Bhayani](https://arpitbhayani.me/blogs/rag-production/)). The LLM-wiki store, being plain Markdown, is **git-diffable** — versioning for free.
- **Access / privacy.** Personal second brains touch private notes; cloud LLM ingestion sends that text to a provider (karpathywiki's privacy escape hatch is fully-local Ollama/LM Studio). Treat any learner-facing input as untrusted (prompt injection — CLAUDE.md).
- **Deletion / forgetting.** RAG makes "unlearning" tractable vs. retraining a model: remove the data from the external store and future queries can't retrieve it ([Milvus](https://milvus.io/ai-quick-reference/how-do-knowledge-graphs-help-in-data-governance)). But beware lag — managed services may retain deleted data for days before purge (Amazon Bedrock KB: ~8 days). The literature distinguishes **deletion** (strong guarantee, costly) vs. **suppression** (cheaper, doesn't fully erase) ([arXiv unlearning](https://arxiv.org/html/2604.12459)). Bi-temporal invalidation is *not* deletion — for a true forget you must hard-remove, not merely expire.

## Operating skills a learner must master

1. **Treat the knowledge store as a living system, not an artifact.** It drifts, rots, and contradicts itself; budget ongoing attention (a periodic `lint`/re-index), don't assume "build once."
2. **Build an eval harness early.** A 30–50-query golden set + faithfulness check, run before each update, is the difference between maintaining and hoping.
3. **Gate updates on eval and escalate conflicts to a human.** Idempotent (hash-skip) re-ingest + regression-gated deploys + human review for contradictions.
4. **Budget for maintenance and for both cost axes** (indexing *and* query), and know your paradigm's specific tax (GraphRAG rebuild cost; LLM-wiki write-time/lint cost; embedding-model lock).
5. **Instrument before scaling.** Tracing with per-request tokens/latency and index-version tags, so you can attribute a quality regression to a cause.

## Caveats

- **Most demos ignore day-2 ops.** Tutorials end at "it answered a question"; they show neither re-ingestion, contradiction handling, nor eval-gated updates — exactly the layer this note (and the curriculum's OPERATE rung) exists to fill.
- **Vendor benchmarks understate maintenance cost.** Quoted indexing prices are one-shot, clean-corpus numbers; they exclude re-indexing churn, dedup/conflict-resolution LLM calls, embedding-model migrations, and human review time.
- **Numbers are directional, not laws.** Cost figures (GraphRAG $20–50/M tokens, 6,000× query spread, 0.1% LazyGraphRAG) come from specific studies/blogs and vary widely with model, corpus, and config — treat as orders of magnitude. Karpathy's anti-RAG framing is an opinion piece, not a measured result (`karpathy-gist.md`).
- **Bi-temporal/graph-memory machinery is young.** Graphiti/Zep are recent and capable but not long-battle-hardened; karpathywiki is ~2 months old with weekly patches and a low bus-factor. Operating-cost claims should be re-checked against current releases.

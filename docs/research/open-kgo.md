# open-kgo — a Python plugin for `mloda` giving one declarative interface over nine families of knowledge-graph backends

- **URL / stars / activity:** https://github.com/mloda-ai/open-kgo — ~6 stars, Apache-2.0, 100% Python. Created 2026-05-22, last pushed 2026-06-20 (~15 commits). Active but very new and tiny; effectively a single-vendor (`mloda-ai`) project, not a community standard.
- **Type:** Library / plugin (a connector framework for an existing data-engineering library, `mloda`). Not a spec, awesome-list, or paper collection.
- **What it is:** open-kgo ("Open Knowledge Graphs and Ontologies") is a plugin for the `mloda` feature-engineering library that abstracts over the knowledge-graph landscape behind one declarative `Feature` interface. You declare what data you want; a backend-specific *reader* fetches it from whatever graph system you point at. It ships nine "connector families" (two concrete readers each, ~18 readers): `network_pg` (Neo4j/Memgraph/Neptune property graphs), `rdf` (SPARQL triple stores via rdflib), `embedded` (NetworkX/igraph/kuzu in-process), `rest_public` (OpenAlex, ConceptNet, STRING), `lineage` (dbt/OpenLineage/DataHub), `code_build` (CycloneDX/SPDX SBOMs), `saas_authz` (OpenFGA/SpiceDB tuple stores), `agent_memory` (Letta/Zep/Mem0), and `citation_rest` (Reactome/OpenAlex). A defining policy: "Every connector and demo runs offline against in-memory libraries or committed fixtures. No Docker, no network."
- **Key ideas / components:**
  - **Reader/connector taxonomy** — the conceptual payload: it argues the "knowledge graph" world is really nine distinct backend families (property graphs, RDF, embedded, public REST, lineage, SBOMs, authz tuples, agent memory, citations) and unifies their query access.
  - **Feature-based declarative access** — request a `Feature`; the framework resolves the right reader. Readers reject multi-feature sets ("single feature per call") to avoid silent mislabeling.
  - **Ontology layer** (`open_kgo.feature_groups.kg.ontology.registry.OntologyRegistry`) — ontologies declared in per-domain YAML (e.g. `metaqa_ontology.yaml`), loaded via an `"ontology"` credential key. Provides edge validation (`is_valid_edge`) and domain/range typing (`get_range_type`) to "catch invalid traversals before they silently return garbage."
  - **Architecture 1 vs Architecture 2 evaluation** — the demos' central experiment: plain traversal (follow any matching edge, no type checking) vs ontology-guided traversal (validate entity type/range before each hop). Includes 1-hop and 2-hop QA harnesses over a hand-authored movie KB derived from the MetaQA dataset. Findings are modest and honest: ontology guidance "**preserves** accuracy while adding per-hop type validation," and the additive ontology layer "does not break existing correct traversals."
  - **Offline-first / fixtures** — testing uses rdflib, networkx, kuzu and committed fixtures rather than live services.
- **Relevance to a modern "second brain" curriculum:** This is squarely the **older symbolic / structured-KG paradigm**, not the LLM-native stack. Repo topics are literally `rdf, ontology, sparql, sbom, knowledge-graph`. The core techniques are SPARQL querying, RDF/OWL-style domain/range ontologies, and graph traversal with type validation — there is **no LLM-driven KG construction, no embeddings, no GraphRAG/vector retrieval, no agentic note-search**. The `agent_memory` family connects *to* LLM-agent memory stores (Letta/Zep/Mem0) but open-kgo itself does not generate graphs from text or do retrieval augmentation. For a rescope it is most useful as (a) a clean, current taxonomy of what "knowledge graph backend" can mean, and (b) a worked, evidence-led example of the symbolic-ontology approach the curriculum is moving *off* — including its honest admission that ontology guidance mainly *preserves* rather than boosts QA accuracy on type-clean data. A builder learning the modern stack would still benefit from the concepts here (property graph vs RDF, ontology domain/range, multi-hop traversal, the agent-memory graph backends), but would pair them with LLM extraction + embeddings + GraphRAG, which this repo deliberately omits.
- **Standout resources / links worth following:**
  - Parent project `mloda`: https://github.com/mloda-ai/mloda and plugin hub `mloda-registry`: https://github.com/mloda-ai/mloda-registry (plus its `docs/guides/`).
  - `demo/eval_arch1_vs_arch2.py` and `demo/eval_qa_accuracy_2hop.py` — the ontology-guided-vs-plain traversal experiments; good template for "does adding structure actually help QA?"
  - `demo/demo_kg_ontology.py` + `open_kgo/feature_groups/kg/ontology/` — minimal YAML ontology + domain/range validation registry.
  - `demo/demo_kg_build_repo.py` — builds an RDF graph from a code repo and queries it via `RdfLibSparqlReader` (concrete RDF/SPARQL ingest example).
  - The nine-family connector taxonomy under `open_kgo/feature_groups/kg/` — a useful map of the KG backend ecosystem (esp. the `agent_memory` family pointing at Letta/Zep/Mem0).
  - MetaQA dataset (eval data): Zhang et al., "Variational Reasoning for Question Answering with Knowledge Graph," AAAI 2018.
- **Caveats:** Very immature and tiny (~6 stars, ~1 month old, ~15 commits, single org). It is a *plugin* for `mloda` — limited standalone value and tightly coupled to that library's `Feature` model. Scope is read/query and validation over *existing* graphs, not construction; the LLM angle is only connectors to agent-memory products. The QA evals run on a small hand-authored fixture with a heuristic question parser, and the authors explicitly caution that hit rates "measure parser coverage and traversal correctness together, not traversal quality alone." Substance over hype, but narrow — treat as a reference taxonomy and a symbolic-stack exemplar, not as a foundation to build the modern curriculum on.

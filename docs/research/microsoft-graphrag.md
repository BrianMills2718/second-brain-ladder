# Microsoft GraphRAG — the reference LLM-built knowledge-graph RAG pipeline (index a corpus into an entity graph + community summaries, then query local-or-global)

- **URL / stars / activity / paper:** [github.com/microsoft/graphrag](https://github.com/microsoft/graphrag) — ~33.9k stars, ~3.6k forks, MIT license. Actively maintained (last push 2026-06-22; shipped on PyPI as `graphrag`). Paper: Edge, Trinh, Cheng, et al., *"From Local to Global: A Graph RAG Approach to Query-Focused Summarization,"* Microsoft Research, [arXiv:2404.16130](https://arxiv.org/abs/2404.16130) (Apr 2024). Companion [MSR blog post](https://www.microsoft.com/en-us/research/blog/graphrag-unlocking-llm-discovery-on-narrative-private-data/). Note: Microsoft labels the repo "demonstration code," not an officially supported product.

- **What it is:** GraphRAG is the canonical, open-source reference implementation of graph-based RAG: a data pipeline that uses an LLM to extract a knowledge graph (entities + relationships) from unstructured text, clusters that graph into a hierarchy of communities, and pre-summarizes each community. At query time it answers either narrowly (local, entity-centric) or globally (map-reduce over community summaries). Its target problem is *global sensemaking* — questions like "What are the main themes in this corpus?" — where vanilla vector RAG fails because the answer isn't in any retrievable chunk; it's a query-focused summarization (QFS) task over the whole dataset.

- **The pipeline (the teachable decomposition):**
  - **1. Text chunking.** Documents are split into "text units" (default ~600 tokens, ~100-token overlap). Chunk size is a real tradeoff: larger chunks mean fewer LLM calls (cheaper) but degraded recall of entities mentioned early in the chunk. GraphRAG mitigates this with self-reflection ("gleaning") prompts that ask the LLM to find entities it missed.
  - **2. Entity + relationship extraction (LLM).** A single LLM prompt over each chunk extracts **entities** (name, type, description), **relationships** (source, target, description, numeric strength), and optionally **claims/covariates** (verifiable factual statements — dates, events, attributes). The prompt is tunable per domain via in-context exemplars (`prompt tune`).
  - **3. Graph construction.** Extracted instances are merged into a graph: duplicate entity mentions are collapsed into nodes, relationships into weighted edges, with descriptions summarized across mentions.
  - **4. Community detection (Leiden).** The Leiden algorithm hierarchically partitions the graph into nested communities of densely-related entities, producing multiple levels (C0 root → leaf communities, e.g. C0–C3).
  - **5. Hierarchical community summarization.** The LLM writes a "community report" (summary) for every community at every level — bottom-up. Leaf reports prioritize high-degree edges and their entity/edge descriptions up to a token budget; higher-level reports either roll up all elements or substitute sub-community summaries when over budget. These summaries are the precomputed answer substrate.
  - **6a. Query time — local search.** Entity-centric. Maps the question to relevant entities, then pulls their neighborhood from the graph (related entities, relationships, covariates) plus the raw text units that mention them. Good for "tell me about X."
  - **6b. Query time — global search.** Theme/sensemaking. Map-reduce over community reports: shuffle and chunk the reports, generate a partial answer + a 0–100 helpfulness score from each chunk in parallel (map), drop score-0 partials, then aggregate the highest-scoring partials into one final answer (reduce). Good for "what are the overarching themes?"
  - **(DRIFT search)** A later hybrid: seeds a local search with community context so the starting point is broader; expands query breadth beyond plain local search.

- **Key concepts a builder must learn:**
  - **Index-time vs query-time split.** The expensive LLM work (extraction + summarization) happens once at indexing; queries reuse the precomputed graph + community reports. This is the central architectural idea.
  - **Local vs global query.** Two distinct retrieval modes for two distinct question shapes (entity lookup vs corpus-wide synthesis) — not interchangeable.
  - **Extraction prompts** as the quality lever: entity/relationship/claim prompts plus gleaning and per-domain prompt tuning.
  - **The graph/community index** (Leiden hierarchy + community reports) as a queryable artifact, distinct from a vector index.
  - **Claims/covariates** as optional structured fact extraction layered on entities.
  - **Cost.** Indexing is LLM-token-heavy: every chunk hits the LLM for extraction, and every community at every hierarchy level gets a generated summary. The repo carries an explicit cost warning; "start small."

- **Relevance to the curriculum:** This is *the* canonical GraphRAG and the natural reference instance for the LLM-native KG/Second Brain spine. Two ideas are directly teachable and reusable in any GraphRAG variant: (1) the **index-time-vs-query-time** split (pay once to structure the corpus, amortize over queries), and (2) the **local-vs-global** distinction (entity neighborhood retrieval vs map-reduce summarization). The paper gives a clean empirical "when it beats vanilla RAG": on global sensemaking questions over ~1M-token corpora, GraphRAG won on **comprehensiveness (~72–83% win rate)** and **diversity (~75–82%)** vs a conventional vector-RAG baseline. The hierarchy levels also expose a cost dial: root-level (C0) summaries used **9×–43× fewer tokens** than mapping over source text at query time, with only modest quality loss, while intermediate levels (C1–C2) gave the best comprehensiveness/cost balance.

- **Caveats:**
  - **Indexing cost & latency.** The big one — an LLM call per chunk plus a generated summary per community per hierarchy level makes indexing slow and token-expensive on large corpora; re-indexing on document change is costly.
  - **Microsoft-stack lean.** Defaults and docs assume Azure OpenAI / OpenAI; configurable to other backends but the happy path is MS-flavored.
  - **Rapid version churn.** Fast-moving repo (config formats, CLI, search methods like DRIFT added over time); officially "demonstration code, not a supported product," so APIs and defaults shift between releases.

## Digest

Microsoft GraphRAG ([arXiv:2404.16130](https://arxiv.org/abs/2404.16130); ~34k-star [repo](https://github.com/microsoft/graphrag), MIT, actively maintained) is the reference graph-RAG system. **Pipeline:** chunk text → LLM-extract entities/relationships/claims → build a knowledge graph → Leiden hierarchical community detection → LLM-summarize every community ("community reports"). **Query time splits two ways:** *local search* (entity neighborhood + raw text units, for "tell me about X") vs *global search* (map-reduce over community reports, for corpus-wide "what are the themes?"). **When it beats vanilla RAG:** global sensemaking over large corpora — ~72–83% comprehensiveness and ~75–82% diversity win rates vs vector RAG, because such answers live in no single retrievable chunk. **Cost caveat:** indexing is LLM-token-heavy (an LLM call per chunk + a summary per community per hierarchy level); Microsoft ships an explicit "start small" warning, and the project is demonstration code with rapid version churn.

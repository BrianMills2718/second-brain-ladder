# Planning a knowledge / second-brain build — the skill of reasoning from your goal to a concrete approach and architecture, neutral across paradigms

- **Scope:** the *planning* skill — the third rung of the learner-empowerment ladder (understand → decide → **plan** → implement → operate). The deliverable here is not a recommended stack; it is the **reasoning discipline** that takes a learner from a stated goal to (a) elicited requirements, (b) a fairly-chosen approach, and (c) a sketched architecture they could hand to the implement rung. A learner who has this skill can plan a build for *their own* goal, and can defend the choice against the alternatives they declined. The paradigms on the menu — vector RAG, GraphRAG (local + global), agent memory (graph vs flat), the LLM-wiki, and plain human PKM — are covered in the sibling notes (`rag-fundamentals.md`, `microsoft-graphrag.md`, `graphrag-jaylzhou.md`, `agent-memory-substrates.md`, `karpathy-gist.md`, `pkm-methods-and-tools.md`); this note is about *choosing among them on purpose*. The single most important framing the curriculum carries through: **PKM and KG/GraphRAG are the same machine described by two cultures** (a `[[wikilink]]` is a graph edge; "AI over my notes" is RAG over a graph — `pkm-methods-and-tools.md`), so the choice is rarely "which is correct" and almost always "which matches my query topology, corpus, and budget."

## Requirements elicitation — the questions that determine the design

The plan starts by interrogating the goal, not the tools. A learner should be able to run this checklist on their own project; each axis maps to a downstream design lever, so requirements thinking *is* the first planning skill. The dominant 2026 framing is explicit: the question is no longer "should we use RAG?" but **"which retrieval architecture matches our query topology?"** ([TianPan](https://tianpan.co/blog/2026-04-19-graphrag-vs-vector-rag-architecture-decision), [nextagile](https://nextagile.ai/blogs/ai/graphrag-vs-rag/)).

- **Corpus — size, type, growth.** How many documents, of what kind (notes, papers, chat logs, code, mixed), and how fast does it grow? Corpus *scale* is a stated GraphRAG driver — the organizational payoff of a graph rises with size, and write-ups put the relationship-graph break-even around corpora that "exceed ~1M documents with incremental updates" ([TianPan](https://tianpan.co/blog/2026-04-19-graphrag-vs-vector-rag-architecture-decision); systematic eval [arXiv:2502.11371](https://arxiv.org/pdf/2502.11371)). A few hundred personal notes is a different machine than a million-token research archive.
- **Question shape — the load-bearing axis.** What will you actually *ask*? The literature converges on three canonical query types ([arXiv:2502.11371](https://arxiv.org/pdf/2502.11371)): **factual / single-hop lookup** ("what's X"), **multi-hop reasoning** ("how is A connected to B across documents"), and **global / sensemaking summarization** ("what are the themes across everything"). RAG is consistently best for single-hop, detail-oriented queries; GraphRAG is advantaged for multi-hop reasoning and corpus-level summaries (the original "Local-to-Global" result: ~72–83% comprehensiveness and ~75–82% diversity win rates on global sensemaking — `microsoft-graphrag.md`, [arXiv:2404.16130](https://arxiv.org/pdf/2404.16130)).
- **Freshness / temporal needs.** How often does the corpus change, and do you ever need to answer "what was true *then*"? Bulk re-embedding is cheap for vector RAG but re-indexing a GraphRAG community hierarchy on every change is costly (`microsoft-graphrag.md`). If facts *evolve and supersede* (preferences, employer, status), you need bi-temporal validity windows — a first-class feature of temporal-KG memory (Graphiti/Zep) and absent from flat stores (`agent-memory-substrates.md`).
- **Privacy / local vs cloud.** Must the data and the model stay on-device? Local-first PKM (Obsidian plain Markdown, Reor + Ollama, Khoj self-hosted) and local-LLM RAG are options; heavy LLM-extraction pipelines (GraphRAG, agentic memory) push you toward cloud spend unless you accept slower local models (`pkm-methods-and-tools.md`).
- **Budget / latency.** What is the per-query and per-index budget, and the latency ceiling? Vector lookup is ~\$0.001/query; agentic retrieval ~\$0.02–0.10 (`rag-fundamentals.md`). GraphRAG is roughly 3–5× the running cost of vector RAG once you count construction and orchestration, and *indexing* a corpus through full MS-GraphRAG has been cited at a 10–40× premium (hundreds of dollars per corpus, with extreme cases at \$33k+) — but newer variants collapse this (LazyGraphRAG <\$5/corpus; HippoRAG 10–30× cheaper than community summarization) ([nextagile](https://nextagile.ai/blogs/ai/graphrag-vs-rag/), [TianPan](https://tianpan.co/blog/2026-04-19-graphrag-vs-vector-rag-architecture-decision)). Strict sub-100ms retrieval favors vector RAG.
- **Single-user vs team.** One person's vault, or a shared/multi-tenant store with isolation, governance, and audit? Multi-user pushes toward managed substrates (Zep, Cognee) with per-user graphs and provenance (`agent-memory-substrates.md`).
- **Artifact: human-readable vs machine-only.** Do you want a durable thing a *human* reads and edits (a wiki, a vault), or is the store purely an index the model queries? This is the cleanest separator between the LLM-wiki / PKM world (compounding, human-legible Markdown — `karpathy-gist.md`) and a pure embedding index (machine-only, re-derived per query).
- **Agency: passive store vs autonomous agent.** Is this a corpus *you* query, or memory an *agent* maintains across sessions (extract → consolidate → re-inject)? The latter is the agent-memory layer, an orthogonal concern from "what store" (`agent-memory-substrates.md`).

## A decision guide (neutral) — map requirements to approach

Present this to learners as a **guide with tradeoffs, not a verdict.** Every row is "if your requirements look like *this*, *these* approaches fit, and here's what you pay." The honest 2026 default is not a single paradigm but a **hybrid baseline** with selective enrichment.

| If your goal / requirements look like… | Candidate approach | Why / the tradeoff |
|---|---|---|
| Small personal corpus, mostly **lookup** ("find the note about X"), human-curated, local, low budget | **Plain PKM** (Obsidian/Logseq + links) or **vector RAG** over the notes | Cheapest, fully local, human-readable; links give navigation. Vector RAG adds semantic "chat with my notes." No graph build needed. The 80% personal case. |
| Mixed corpus, **factual + semantic** lookup, needs exact-match on names/codes too | **Hybrid: vector + BM25 + reranker** | The strong production default — ~15–25% accuracy lift over plain vector "with no graph infrastructure required" ([TianPan](https://tianpan.co/blog/2026-04-19-graphrag-vs-vector-rag-architecture-decision)); recovers exact-match recall vectors miss (`rag-fundamentals.md`). |
| Cross-cutting **"what are the themes / overarching patterns"** over a large corpus | **GraphRAG global search** | Answers live in *no single chunk*; map-reduce over community summaries wins on comprehensiveness/diversity (`microsoft-graphrag.md`). Pay the indexing premium. |
| **Multi-hop** "how is A connected to B across documents"; relationship-heavy domain | **GraphRAG local / graph traversal** (or HippoRAG-style) | Graph gives the relational context vector similarity can't; flagged especially when recall@k <60% on representative multi-hop queries ([TianPan](https://tianpan.co/blog/2026-04-19-graphrag-vs-vector-rag-architecture-decision)). |
| An **agent** that must remember facts/preferences across sessions; facts mostly independent | **Flat / extracted-fact agent memory** (Mem0, Memobase) | Cheapest, fastest agent-memory path; "what do I know about X" without multi-hop (`agent-memory-substrates.md`). |
| Agent memory where **relationships matter and facts change over time** (need "true then") | **Graph agent memory** (Graphiti/Zep, Cognee) | Bi-temporal validity + provenance + multi-hop; cost is extraction spend and operational complexity (`agent-memory-substrates.md`). |
| Want a **durable, human-readable artifact** that compounds, plus an agent that navigates it | **LLM-wiki** (Karpathy pattern) | Moves synthesis to *write* time; retrieval becomes navigation of pre-linked, pre-deduped pages. Strong north-star, but unbenchmarked, and adds maintenance/drift risk (hence `lint`) — `karpathy-gist.md`. |
| Honest **starting point** for most builders unsure of query topology | **Hybrid baseline now, instrument, enrich later** | Measure recall@k/precision@k by query type *before* committing to a graph; add graph only where the metrics fail (phased sequencing below). |

The guide's posture: these are **not mutually exclusive**, and the live trend is composition (vector for local, graph/global for exploratory; flat memory with optional graph backend). The wrong move is picking a paradigm by hype; the right move is reading your own requirements off the checklist and matching.

## Sketching the architecture — the components to lay out

Whatever approach is chosen, a second-brain build instantiates the **same component skeleton.** A good plan names each box and says how the chosen approach fills it. Teaching the skeleton (rather than one stack) is what lets a learner re-plan when the menu shifts.

1. **Ingestion** — get sources in and chunk/parse them. *Vector RAG:* chunk + embed. *GraphRAG:* chunk → LLM entity/relationship/claim extraction (`microsoft-graphrag.md`). *Agent memory:* capture turns → LLM extract/consolidate facts. *LLM-wiki / PKM:* read source → write/update human-readable pages; capture into a vault.
2. **Store** — where knowledge durably lives. Vector DB (FAISS/Chroma/pgvector/Qdrant), a knowledge graph (Neo4j/triple store), pinned text blocks, structured profiles, or plain Markdown files. The artifact axis decides whether this is machine-only or human-legible.
3. **Index** — what makes the store fast/queryable. ANN/HNSW vector index, BM25 lexical index, a Leiden community hierarchy + community reports (GraphRAG), a PPR matrix, or an `index.md` / MOC (LLM-wiki, PKM).
4. **Retrieval** — the query-time policy. Keyword / vector / hybrid+rerank / graph operators (the 16-operator decomposition in `graphrag-jaylzhou.md`) / local-vs-global / agentic. This is the box where the chosen approach is most visible; treat retrieval as *operator composition*, not a monolith.
5. **Generation** — assemble context under a token budget and produce the grounded, cited answer (mind lost-in-the-middle; place strongest evidence at the edges — `rag-fundamentals.md`).
6. **UI / agent** — how the human or agent interacts: a chat box, an Obsidian vault, a wiki navigator, or an autonomous agent that pages memory in/out (MemGPT/Letta tier — `agent-memory-substrates.md`).
7. **Eval** — measure retrieval and generation *separately*: recall@k / MRR for retrieval; faithfulness / answer-relevance (RAGAS) for generation (`rag-fundamentals.md`). The plan must say what "good enough" means *before* building, because the metrics gate the enrichment decision.
8. **Update path** — how new sources land and how stale/contradictory knowledge is reconciled: re-embed (cheap), re-index the graph (costly), consolidate/dedup facts, supersede with timestamps, or `lint` the wiki for contradictions and orphans. The freshness/temporal requirement decides how heavy this box is.

## Planning skills a learner must master

- **Requirements thinking** — interrogate the *goal* before the tools; run the elicitation checklist and let query shape, corpus, freshness, privacy, budget, and audience drive everything. The recurring novice failure is choosing a paradigm first and reverse-justifying it.
- **Mapping needs → options** — read requirements off the checklist and onto the decision guide, *and articulate the alternatives declined and why.* Fairness across paradigms is the skill; a plan that can't say what it isn't doing hasn't reasoned.
- **Sizing / cost estimation** — back-of-envelope the index cost, per-query cost, and latency before committing. Know the order-of-magnitude facts (vector ~\$0.001/q; GraphRAG 3–5× running, 10–40× indexing premium but variants collapse it; agentic \$0.02–0.10/q) so the budget axis is quantitative, not vibes.
- **Sequencing a build (MVP first)** — phase it: (1) instrument and measure recall@k/precision@k by query type, (2) ship a hybrid baseline (vector + BM25 + reranker), (3) add selective graph enrichment only where Phase 2's metrics fail, (4) reserve full GraphRAG/global for corpora that genuinely need synthesis ([TianPan](https://tianpan.co/blog/2026-04-19-graphrag-vs-vector-rag-architecture-decision)). Build the measurement before the machinery.
- **Reversibility / cost-to-change** — distinguish cheap-to-change from sticky decisions. Plain Markdown + links is maximally portable (the reason the wiki and PKM worlds insist on it — `pkm-methods-and-tools.md`); a bespoke graph schema and a community-summary index are expensive to undo. Prefer the decision you can walk back, and keep an exit format (portable Markdown, exportable store) so today's choice doesn't trap tomorrow's.

## Caveats

- **Premature optimization is the default failure.** "Start small" is the explicit warning on the canonical GraphRAG repo (`microsoft-graphrag.md`); most personal builds never need a graph. Ship the simplest thing that answers your real questions, measure, and enrich only on evidence. A graph you built before you had a multi-hop query is cost with no payoff.
- **The menu shifts as models and tooling improve.** Long-context windows, cheaper LLM-built graphs (LazyGraphRAG, HippoRAG), and better agentic routers move the break-evens quarterly; the recurring "RAG is dead" claim is best read as "*naive* RAG is dead" — retrieval became a conditional policy, not obsolete (`rag-fundamentals.md`). Anchor the plan to the **durable component skeleton and the requirements reasoning**, treat specific tools and cost figures as perishable, and re-run the decision guide when the inputs change.

## Digest

The planning skill turns a goal into requirements → approach → architecture, neutral across paradigms. **Requirement axes:** corpus (size/type/growth), question shape (factual vs multi-hop vs global summarization — the load-bearing axis), freshness/temporal, privacy/local-vs-cloud, budget/latency, single-user-vs-team, human-readable-vs-machine artifact, and passive store vs autonomous agent. **Neutral requirements→approach map:** small notes + lookup → plain PKM or vector RAG; mixed factual+semantic → hybrid (vector+BM25+rerank, ~15–25% lift, the honest default); "what are the themes" → GraphRAG global; multi-hop relationships → GraphRAG local/traversal; agent recall of independent facts → flat memory (Mem0); evolving/temporal/related facts → graph memory (Graphiti/Zep); durable human-readable + agentic navigation → LLM-wiki — and these compose rather than exclude. **Architecture components:** ingestion, store, index, retrieval (operator composition, not monolith), generation, UI/agent, eval (recall@k/MRR + faithfulness, measured before building), update path. **Planning skills:** requirements thinking, mapping needs→options (and defending alternatives declined), sizing/cost estimation, MVP-first sequencing (instrument → hybrid baseline → selective graph → full GraphRAG only on evidence), and reversibility (favor portable Markdown and cheap-to-undo choices). **Caveats:** premature optimization is the default failure ("start small"); the menu shifts as long-context and cheap graph variants move the break-evens, so anchor to the skeleton and the reasoning, not the tools.
